# Offline PWA Support — Design Document

## Overview

Kanbeasy is a client-side kanban board that already stores all data in localStorage. It is a natural fit for a Progressive Web App: the entire application is a static SPA with zero server-side data fetching. Adding PWA support makes the app installable on desktop and mobile, ensures it works fully offline (even for first-time visits after initial load), and gives users a native-app feel with standalone window, home screen icon, and update notifications.

### Goals

1. Make kanbeasy installable via browser install prompts on Chrome, Edge, Firefox, and Safari (iOS/macOS).
2. Cache all assets so the app works offline after the first visit.
3. Provide a clear update flow when new versions are deployed to GitHub Pages.
4. Pass the Lighthouse PWA audit at 100%.
5. Keep the dependency footprint minimal per project conventions.

### Non-Goals (Phase 1)

- Background sync, push notifications, or any server-side features.
- Multi-device data synchronization.
- Native app store distribution.

## Web App Manifest

A new file `manifest.webmanifest` will be generated by `vite-plugin-pwa` at build time from configuration in `vite.config.ts`. The manifest is not hand-authored as a static file.

### Specification

```json
{
  "name": "Kanbeasy — Simple Kanban Board",
  "short_name": "Kanbeasy",
  "description": "An ultra simple task organizer. Drag-and-drop kanban board with offline support, themes, analytics, and local-only data storage.",
  "start_url": ".",
  "scope": ".",
  "display": "standalone",
  "orientation": "any",
  "theme_color": "#2d1b4e",
  "background_color": "#2d1b4e",
  "categories": ["productivity", "utilities"],
  "icons": [
    { "src": "icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
    { "src": "icons/icon-512.png", "sizes": "512x512", "type": "image/png" },
    {
      "src": "icons/icon-maskable-512.png",
      "sizes": "512x512",
      "type": "image/png",
      "purpose": "maskable"
    }
  ],
  "shortcuts": [
    {
      "name": "New Card",
      "short_name": "New Card",
      "url": "?action=new-card",
      "icons": [{ "src": "icons/icon-192.png", "sizes": "192x192" }]
    }
  ]
}
```

### Design Decisions

**`display: standalone`** over `minimal-ui`. Standalone provides the most native feel — no browser URL bar. Since kanbeasy has no navigation that requires a URL bar, standalone is appropriate. Users can always access browser features via the three-dot menu.

**Static `theme_color` and `background_color`**. The manifest is baked at build time, so these cannot follow the user's active theme at install time. Using the Twilight dark purple (`#2d1b4e`) matches the existing `kanban.svg` icon background color and provides a cohesive look on splash screens and the title bar. The `<meta name="theme-color">` tag in `index.html` can still be dynamically updated at runtime by the ThemeProvider to match the active theme (this affects the browser/OS chrome color while the app is running).

**`start_url: "."`**. This works correctly with the GitHub Pages base path (`/kanbeasy/`) because vite-plugin-pwa resolves it relative to the configured `base`.

## Icon Requirements

### Source

The existing `kanban.svg` at the project root is already 512x512 and serves as the source icon. It features a purple rounded-rect background with three kanban column bars in varying heights and opacities.

### Required Generated Icons

| File                          | Size    | Purpose                                |
| ----------------------------- | ------- | -------------------------------------- |
| `icons/icon-192.png`          | 192x192 | Standard PWA icon (Android, desktop)   |
| `icons/icon-512.png`          | 512x512 | High-res PWA icon, splash screen       |
| `icons/icon-maskable-512.png` | 512x512 | Maskable icon (Android adaptive icons) |
| `icons/apple-touch-icon.png`  | 180x180 | iOS home screen                        |
| `favicon.ico`                 | 32x32   | Legacy favicon fallback                |

### Maskable Icon Considerations

The existing SVG design has safe-area compatibility because the kanban bars are centered within the 512x512 canvas with generous padding (80px left, 80px right, 96px top, 96px bottom). The maskable safe area is the inner 80% circle (roughly 410px diameter centered). All three bars fall well within this zone. The maskable variant should use the same design, possibly with a slightly enlarged background fill to ensure the rounded corners are fully cropped by the mask without exposing any transparency.

### Generation Approach

Use a one-time script (not a runtime dependency) to generate PNGs from the SVG source. Options:

- `sharp` (Node.js, zero native deps): `sharp(svg).resize(192).png().toFile(...)` — lightweight, already in the Node ecosystem.
- Manual generation via any SVG-to-PNG tool (Inkscape CLI, Figma export, etc.).

These generated PNGs are committed to the repo under a new top-level `icons/` directory. Committing them is simpler and avoids adding `sharp` as a dev dependency.

### Dynamic Favicon Interaction

The app currently generates a dynamic favicon at runtime via `src/theme/favicon.ts` using a data URI SVG. This is unrelated to the PWA manifest icons and will continue to work. The static `kanban.svg` referenced in `index.html` serves as the initial favicon before React hydrates. After PWA install, the OS uses the manifest icons for the app launcher; the dynamic favicon only matters in browser tabs.

A new `<link rel="apple-touch-icon">` tag should be added to `index.html` pointing to `icons/apple-touch-icon.png`.

## Service Worker Strategy

### Plugin Choice: `vite-plugin-pwa`

**Recommendation: Use `vite-plugin-pwa`.**

Evaluation against project dependency criteria:

| Criterion           | Assessment                                                                                                                                                                                                                                                                                                  |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Bundle size**     | Zero runtime bundle impact. The plugin generates the service worker and manifest at build time. The only runtime code is a small SW registration snippet (~200 bytes).                                                                                                                                      |
| **Maintenance**     | Actively maintained (antfu/vite-plugin-pwa), 3k+ GitHub stars, regular releases, responsive issues.                                                                                                                                                                                                         |
| **Transitive deps** | Pulls in `workbox-build` and related Workbox packages as dev dependencies only. These do not ship to the client.                                                                                                                                                                                            |
| **Necessity**       | Hand-rolling a service worker is feasible for this simple app, but vite-plugin-pwa handles: manifest injection, precache manifest generation with content hashes, SW registration with update detection, and Workbox strategy configuration. Replicating this manually would be ~200+ lines of boilerplate. |

The plugin is a **dev dependency only** — it adds zero bytes to the client bundle beyond the generated SW file itself.

### Alternative Considered: Hand-Rolled Service Worker

A minimal hand-rolled SW would use the Cache API to precache a hardcoded list of assets. The downsides:

- Must manually update the asset list on every build (or parse the Vite manifest).
- No automatic content-hash-based cache busting.
- Must implement update detection and cache cleanup from scratch.
- Workbox precaching handles edge cases (partial responses, opaque responses, quota management) that are tedious to implement correctly.

For an app that will continue to grow, the plugin's automation is worth the dev dependency.

### Caching Strategy

Since kanbeasy has no API calls and all data lives in localStorage, the caching strategy is straightforward:

**Precache everything.** All built assets (JS chunks, CSS, SVG, HTML, icons) are precached at SW install time. The Workbox `generateSW` mode handles this automatically by reading the Vite build manifest.

```typescript
// vite.config.ts PWA plugin configuration
VitePWA({
  registerType: "prompt", // show update prompt, don't auto-reload
  workbox: {
    globPatterns: ["**/*.{js,css,html,svg,png,ico,woff2}"],
    cleanupOutdatedCaches: true,
    clientsClaim: true,
  },
  manifest: {
    /* ... manifest config from above ... */
  },
  includeAssets: ["kanban.svg", "icons/*.png"],
});
```

There is no need for network-first or stale-while-revalidate strategies because there are no network requests to intercept beyond the initial asset load. Once cached, the app serves entirely from cache.

### Navigation Fallback

The SW should serve `index.html` for all navigation requests (SPA fallback). Workbox's `navigateFallback` handles this:

```typescript
workbox: {
  navigateFallback: 'index.html',
  navigateFallbackDenylist: [/^\/api/],  // future-proofing
}
```

## Update Flow

### Strategy: Prompt-Based Update

When a new version is deployed to GitHub Pages, the service worker detects updated assets during its periodic check (or on page load). The user is shown a non-intrusive prompt to update.

### Implementation

1. **Detection**: `vite-plugin-pwa` with `registerType: 'prompt'` exposes a `useRegisterSW` virtual module that provides `needRefresh` and `updateServiceWorker` callbacks.

2. **UI Component**: A new `UpdatePrompt.tsx` component renders a small banner at the bottom of the screen (above the BottomBar) when an update is available.

```
+----------------------------------------------------+
|  A new version is available.  [Update]  [Later]    |
+----------------------------------------------------+
```

Styling follows existing patterns: uses `tc.border`, `tc.glass`, `tc.text` class helpers. Positioned fixed at the bottom, similar to the BottomBar pill style.

3. **Behavior**:
   - "Update" calls `updateServiceWorker(true)` which activates the new SW and reloads the page.
   - "Later" dismisses the banner for the current session. The prompt reappears on next page load if the update is still pending.
   - No data loss risk: all data is in localStorage, which persists across SW updates and page reloads.

4. **Registration**: In `src/main.tsx`, register the SW after React renders using the React-specific `virtual:pwa-register/react` module that provides a `useRegisterSW` hook.

### SW Update Check Frequency

Browsers check for SW updates on navigation and roughly every 24 hours. For a more responsive update experience, the app can check on visibility change (when the user returns to the tab):

```typescript
document.addEventListener("visibilitychange", () => {
  if (document.visibilityState === "visible") {
    updateSW(); // check for updates
  }
});
```

## Offline Indicator

Since kanbeasy works fully offline by design (all data is local), a persistent offline indicator is unnecessary and potentially confusing. **Recommendation**: Do not add an offline indicator in Phase 1. If a future "sync" or "share" feature requires network access, add an indicator at that time.

## Install Experience

### Browser Default Prompts

Modern browsers (Chrome, Edge) show an install icon in the address bar when a valid PWA manifest is detected. This is sufficient for Phase 1 — no custom install UI is needed initially.

### Custom Install Banner (Phase 2)

For a better conversion rate, a custom install banner can be shown by intercepting the `beforeinstallprompt` event:

1. Listen for `beforeinstallprompt` in a `useInstallPrompt` hook.
2. Store the event and show a banner after the user has dismissed the welcome modal and interacted with the board at least once (avoid overwhelming first-time users).
3. Show: "Install Kanbeasy for quick access from your home screen. [Install] [Not now]"
4. "Not now" sets a localStorage flag to suppress the prompt for 30 days.

### iOS Safari Considerations

Safari does not fire `beforeinstallprompt`. iOS users must manually use "Add to Home Screen" from the share sheet. The custom install banner should detect iOS Safari and show instructions: "Tap the Share button, then 'Add to Home Screen'." This is Phase 2 work.

### Meta Tags for iOS

Add to `index.html`:

```html
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta
  name="apple-mobile-web-app-status-bar-style"
  content="black-translucent"
/>
<meta name="apple-mobile-web-app-title" content="Kanbeasy" />
<link rel="apple-touch-icon" href="/icons/apple-touch-icon.png" />
```

## Performance and Lighthouse

### Required for Full PWA Compliance

1. Valid web app manifest with required fields (name, icons, start_url, display).
2. Service worker that controls the page.
3. All content served over HTTPS (already true on GitHub Pages).
4. `<meta name="theme-color">` tag in HTML.
5. Maskable icon in the manifest.
6. Redirects HTTP to HTTPS (handled by GitHub Pages).
7. Configured for a custom splash screen (manifest with `background_color`, `theme_color`, `name`, and 512x512 icon).

### Lighthouse Config Update

Add PWA assertion to `lighthouserc.json`:

```json
{
  "categories:pwa": ["warn", { "minScore": 0.9 }]
}
```

Use `"warn"` initially until the PWA score is stable, then promote to `"error"`.

## Dependencies Evaluation

### `vite-plugin-pwa` (recommended)

- **npm package**: `vite-plugin-pwa`
- **Dev dependency only**: Yes — zero client bundle impact
- **Transitive deps**: `workbox-build`, `workbox-core`, `workbox-precaching`, `workbox-routing`, `workbox-window` (all dev-only, used to generate the SW at build time)
- **Maintenance**: Actively maintained by antfu (Anthony Fu), part of the vite-pwa org. Regular releases, responsive to issues.
- **Bundle size**: The generated service worker file is typically 5-15 KB (gzipped). The registration snippet added to the app is ~200 bytes.
- **Gzipped size of npm package**: ~35 KB (dev dependency, does not ship to client)

### No Other Dependencies Needed

The plugin provides everything: manifest generation, SW generation, registration utilities, and the React hook for update detection.

**Steps:**

1. Install `vite-plugin-pwa` as a dev dependency.
2. Generate PNG icons from `kanban.svg` (one-time, commit to repo).
3. Configure the plugin in `vite.config.ts` with manifest, workbox, and `registerType: 'prompt'`.
4. Add meta tags to `index.html`.
5. Create `UpdatePrompt.tsx` using the `useRegisterSW` hook from `virtual:pwa-register/react`.
6. Add `UpdatePrompt` to `App.tsx`.
7. Add TypeScript type declarations for the virtual PWA module.
8. Update `lighthouserc.json` with PWA assertion.
9. Run `npm run kitchen-sink` to validate everything passes.

### Phase 2: Enhanced Install Experience

- Custom install banner with `beforeinstallprompt` interception.
- iOS Safari "Add to Home Screen" instructions.
- Suppress prompt for 30 days after dismissal (new localStorage key in `STORAGE_KEYS`).

### Phase 3: Advanced PWA APIs

These are optional enhancements, each independent:

- **Manifest shortcuts**: "New Card" shortcut (already specced in manifest above). Requires handling `?action=new-card` query param in `App.tsx` to auto-focus the first column's add card input.
- **Share Target API**: Add `share_target` to manifest so users can share text from other apps into kanbeasy as a new card title.
- **File Handling API**: Register kanbeasy as a handler for `.json` files so board exports can be opened directly.
- **Badge API**: Show a count badge on the app icon (e.g., cards in progress). Limited browser support; nice-to-have.
- **Dynamic theme-color**: Update `<meta name="theme-color">` from `ThemeProvider.tsx` when the theme changes, so the OS title bar matches the active theme in standalone mode.

## Open Questions

1. **Icon generation workflow**: Should icons be generated once and committed, or generated at build time via a script? **Recommendation**: Commit the PNGs. They change rarely, and it avoids a native dependency.

2. **Base path handling**: The app uses a dynamic `config.base` from `PUBLIC_URL` env var (for GitHub Pages). Verify that `vite-plugin-pwa` correctly resolves manifest URLs and SW scope relative to this base.

3. **SW scope on GitHub Pages**: The app is deployed to `https://darrenjaworski.github.io/kanbeasy/`. The SW scope will be `/kanbeasy/` which is correct. Verify that `navigateFallback` resolves to the correct path.

4. **Dev mode behavior**: `vite-plugin-pwa` supports a `devOptions` config to enable the SW in dev mode for testing. **Recommendation**: Enable with `devOptions: { enabled: true }` during development.

5. **Update prompt UX timing**: Should the update prompt appear immediately on detection, or wait until the user is idle? **Recommendation**: Show immediately — it is non-modal and positioned out of the way.

6. **Dynamic theme-color meta tag**: Should the `ThemeProvider` update `<meta name="theme-color">` on theme change in Phase 1 or Phase 3? **Recommendation**: Include in Phase 1 — small addition (3-4 lines) that improves the standalone experience.

7. **Feature flag**: Should PWA registration be behind a feature flag? **Recommendation**: No — it adds no runtime cost when the browser does not support SW, and it only improves the experience.
